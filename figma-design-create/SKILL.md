# Figma 新規デザイン作成の作業標準

**TL;DR**: 新規画面/フロー/フレームを既存DSの上に要件から起こすときに発火。出力は《要件確定 → ビューインベントリ＋ページIA宣言 → 現物確認 → DS棚卸し → 設計＋ゲート → ユーザー一括承認 → 段階実装 → プロトタイプ配線 → 全件レビュー → 自分の目で最終確認》を経た新規デザインと配線。実行は全部サブエージェントに委譲し、**自分はマネジメント（要件確定・割当・ゲート判定・全件カバレッジ確認・完了可否）だけ**を握る。

## 適用範囲と境界

### やる / やらない
- **やる（本スキル）**: 0→1で画面/フレームを起こす作業（新規UIの設計・実装・既存フローへの配線）。
- **やらない（委譲）**: 既存frameへのテキスト追加・行追加・バリアント差し替え・トークンバインド是正等の**既存改修**は `figma-ds-edit`。新規Component（atom/molecule）設計・property軸の大改修は `figma-component-design`。体験のE2Eテストは `figma-e2e-test`。`use_figma` 実行の低レベル作法は `figma-use`。
- **入口**: Figma作業はまず `mikeneko-figma` を通り、共通禁止事項（Spacer禁止/手描き偽装/アンチブリード=default-deny/AA・14px/出典ゲート/OOUI原則/ページ編成原則）の正典（SOT）を確認する。本スキルは自分のゲートで実検証する項目だけを自足の完全文で持つ。

### 適用分岐（新規 / 既存改修 / 混在）を前方確定
- **新規作成のみ**: 本スキルで完結。
- **既存改修のみ**: `figma-ds-edit` へ。
- **混在運用（新規＋既存改修）**:
  1. **マネージャー自身がスキル呼出主体**。本スキル§0で混在リスト（新規／改修を分離）を宣言。
  2. 本スキル§1〜§3を進める間、改修分は**並行で `Skill(figma-ds-edit)` 起動**して棚卸し・現物確認を実施。**ds-edit の棚卸し結果は本スキル§2の入力として引き継ぐ**（二重走査しない）。
  3. **§4ユーザー一括承認は新規分＋改修分のマッピング表をまとめて1メッセージで提示**。**ds-edit内で発生したエスカレ票（新規variant、新規variable、State B'等）も本スキルマネージャーが吸い上げて一括承認に束ねる**（ds-edit側で勝手に承認を取らない）。
  4. §5実装は新規分（本スキル）と改修分（ds-edit）を**領域別並列**で進める（同一ファイル同時書き込みは衝突回避のため避ける）。
  5. **§6プロトタイプ配線は本スキルにのみ実施**（ds-edit側の配線節は無視、二重実行禁止）。**既存frameへのreaction張りは「改修」扱いとして§0改修対象リストに含める**（reactionは既存ノードへの変更なので改修スコープ監査の対象）。
  6. §7レビュー・§8自分の目は新規＋改修両方を対象。

## 鉄則（不可侵原則・最優先・全工程共通）
**第1ランク（不可侵・default-deny）**: 鉄則2 出典必須 ／ 鉄則4 生値禁止 ／ 鉄則13 OOUI=オブジェクトのビュー。番号は固定（他§のクロス参照を保つため付け替えない）。

1. **要件をまず理解する。** 着手前に「画面の役割／満たすべき条件／曖昧な分岐」を一文化し、**作成フレームリスト・改修対象リスト・遷移配線リスト**まで宣言する。リスト外は作らない。曖昧があれば実装前にユーザーへ確認するが、**質問は規模に応じて束ねる**（後述 S7「規模スケーリング」）。
2. **要件にない文言を加えない。** 使ってよいのは **(a) 要件に明示されたコピー** ／ **(b) 既存DS・既存画面に実在するコピー** ／ **(c) 機能必須要素（後述ホワイトリスト）** ／ **(d) ダミーデータ（`placeholder:`タグ付き）** のいずれか。各テキストに**出典タグ**（`req:` / `ds:<id>` / `existing:<nodeId>` / `inherent:` / `placeholder:` / `guess:` のいずれか）を残す。`guess:` が3件以上を占めるならスコープ自体を要件主に確認。
3. **判断点は相談する。質問は束ねる。** 追加文言／新規変数／新規スタイル／新規コンポ／新規パターン等、自分で決められない分岐は実装前にユーザーへ4択 or 明確な質問で上げる。「よくあるやつ」「いい感じ」等の曖昧指示には9項目テンプレを丸投げせず、**最小コア版＋拡張候補リスト**の2点セットで確認（後述 S7「曖昧指示の扱い」）。
4. **生値を残さない。** 直書きhex・font-size・line-height・font-family・letter-spacing・spacing・radius・elevation 禁止。色は役割→セマンティック変数、タイポはテキストスタイル or タイポ変数を必ずバインド。**例外**: DS不在ルートで primitive 変数を**初回定義する瞬間のみ**生値OK（設計マッピング表に `primitive-def:` タグで明示）。
5. **新規バリアブル/スタイル/コンポは無断で作らない。** 棚卸しで既存に類似が無いと**全Nページ走査済の上で**確定したうえ、命名・所在コレクション・スコープ・参照元込みでユーザーへエスカレーション。承認まで生値で進めず保留。**ファイルが空・DS未着手のときは「ファイル空・DS未定義」の一行で代替してよい**（儀礼スクショ不要）。
6. **既存DS流用は命名一致では決めない。** 候補が見つかったら、**既存使用箇所を最低1件取得して用途一致を検証**してから採用。State A（完全一致）／State B（部分一致・要承認）／**State B'（合成内部欠落・要承認）**／State C（不採用＝新規必要）の4分類（後述 S7「DS流用判定」）。
7. **発見ファースト・決め打ち禁止。** トークン名・コンポーネント・ノードIDはファイルごとに違う。検索/一覧の空振りは不在の証拠でない。
8. **段階的に作る、一気にやらない。** 1画面を1回の `use_figma` で完成させない。論理ブロック単位（コンテナ→骨格→セクション中身→テキスト確定→**プロトタイプ配線**）で分割。各段でスクショ自己確認＋自分のゲート。
9. **use_figma の atomic 性を信じる、レスポンス検証は毎回。** 失敗時はスクリプト全体が未実行。エラー瞬間に停止し、現状を `get_screenshot` で確認、原因特定してから次へ。連投禁止。
10. **見た目が部品なら実体も部品。手描き偽装禁止。** タブ／入力／バッジ／ボタン／チップ／メニュー行など DS に実体がある部品を FRAME+TEXT で手描きしない。必ず instance 化。
11. **検証はサンプルでなく全件被覆。** 配置した全要素＋全配線を見る・node type が `INSTANCE` か・mainComponent が想定通りか・detach されていないか・fill/typo がバインド済みか・**reactionが正しく張られているか**、を1件ずつ。
12. **報告は正直に。** 残課題・未承認の保留事項を隠さない。先回りの正当化や未検証の約束をしない。
13. **各フレームはオブジェクトのビューとして起こす。** 画面は「何のオブジェクト(名詞)の・どのビュー(collection一覧/single詳細)か」を先に確定してから作る。動詞(作成/編集/日程調整/定期化/リマインド)はビュー上のモードレスなアクションchip・overlay・inline編集にし、独立画面・ステップ・ウィザードにしない。種別/立場/日程/定期/リマインドはオブジェクトでなくプロパティ・状態・アクション。例外(逐次タスク)はproc:理由コード宣言制。

## 標準オーダー（0→9）
工程番号はこの1系統のみが正本（ロールの丸数字は S5、対応はレジェンド参照）。

- **0. 要件確定〔委譲不可・自分〕** — S7「要件理解の手順」に沿い、**規模クラス・作成フレームリスト・改修対象リスト・遷移配線リスト・OOUIビューインベントリ（表A/表B）・ページIA宣言（表C/表D）**まで含めて1枚にし、ユーザー一括confirm を取る。**ここが固まらない限り着手しない。**
- **1. 現物確認＋既存コピー横断走査（読み取り）** — 同じファイル内の**既存類似画面**を `get_screenshot`＋`get_metadata` で観る。さらに**今回扱うラベル候補について全ページのテキスト走査**を行い、既存表記を確認（表記揺れを増やさない）。**ファイルが完全に空のときは「空でした」の1行宣言で完了**とし、儀礼スクショは不要。
- **2. 棚卸し（読み取り）** — 既存コンポーネント／変数／テキストスタイルを全走査で積極列挙（姉妹スキル `figma-ds-edit` の棚卸し runbook を流用）。**ファイル空時は「DS未定義」の1行で完了**。
- **3. 設計＋ゲート判定〔委譲不可・マネージャー〕** — 画面構造／使うコンポーネントID＋バリアント／使う変数・テキストスタイル／**全テキストの文言と出典**／**遷移配線リスト**を**設計マッピング表**（フォーマットは S7、改修行・繰り返し要素も含める。frame行に object / view(collection|single|proc:<code>) / modeless-action 列を追記）にする。①新規文言、②新規変数/スタイル、③新規コンポ、④既存と異なる扱い、⑤機能必須として追加した要素、**⑥ OOUI逸脱(動詞フレーム/種別選択画面/ウィザード化/独立日程・定期フレーム)** の6種は**エスカレーション対象**として切り出す。
- **4. ユーザー一括承認** — 3 の設計＋エスカレーション項目を**1メッセージにまとめて**提示（L規模は領域別）。**改修分のマッピング表も同じ承認に含める**（混在運用時）。**ビューインベントリ（表A/B）・ページIAマニフェスト（表C/D）・`proc:`/`cross-object:`/`non-object:` 例外フレーム＋各例外の反証可能な正当化1行も同じ承認に束ねる**。**auto-layout既定から外す `abs:<code>` 例外＋各々の『なぜgap/paddingでは表現できないか』理由1行も同じ承認に束ねる**（新たな承認往復を増やさない）。承認前に書き込みしない。
- **5. 実装（書き込み・段階）** — `/figma-use` 遵守。1段＝1論理ブロック。各段で自己スクショ→レスポンス検証→OKなら次。同一ファイル同時書き込み禁止（**ただし重ならない領域は並列可**）。
- **6. プロトタイプ配線** — §0の遷移配線リストに従って、新規/既存問わず **Reaction (On click → Navigate to)** を設定する専用段。**遷移配線リストが空（1画面案件）の場合は「対象なし」で完了宣言してよい**。**配線後に OOUI逆監査(f) の reactionグラフ部を再実行する（配線前は reaction 未取得＝判定不能）。**
- **7. 独立レビュー（読み取り・全件）** — 実装者と別個体・敵対的。レイアウト・コピー・変数バインド・instance実体・**reaction整合・object/view整合・modeless退行（動詞フレーム化やウィザード化）**を全件で確認。
- **8. 自分の目で最終確認〔自分のゲート〕** — マネージャー直接実施。**具体手順**: (a) 作成フレーム全数を `get_screenshot` で目視 (b) 主要instance を `get_metadata` で `INSTANCE` / mainComponent 一致 / detach無しを抽出確認 (c) 遷移配線リストN件と実装reaction件数のN突合 (d) 設計マッピング表の各行が実装と一致。**件数突合は cardinality 対応式**: 設計N≦実装N＝レビューN（template 行は設計1で許容、実装＝レビューは厳密一致）、配線N＝reactionN（こちらは厳密一致）。(e) **逆差分監査（実装→宣言）**: 実装に存在する全要素/フィールドを列挙し、各々が§0作成リスト/§3マッピング表に宣言済みかを逆向きに突合。宣言に無い要素（template 繰り返しを除く）が出ていたら**ブリード/スコープ膨張として削除かエスカレーション**（→ S7「要素・フィールドの出典とアンチブリード」）。(f) **OOUI逆監査（実装→宣言、(e)逆差分監査の直後・判定者は実装者と別個体＝§7敵対レビュー）**: 実装済み全top-level frameを列挙し、表Bへの写像・view種別の構造裏取り・modeless退行シグナル・reactionグラフの線形ウィザード検出・collection⇄single往復到達を機械突合（詳細は S7「OOUI逆監査」）。**§6配線前はreaction未取得＝判定不能、§6配線後に再監査**。(g) **ページ編成逆監査**: object→Page集合を実測再構築し、散在/孤立/帰属/横断の4検査（詳細は S7「ページ編成逆監査」）。(h) **auto-layout厳守逆監査（実装→宣言・判定者は実装者と別個体＝§7敵対レビュー）**: §0表Bの『auto-layout既定＋abs:例外宣言』に対し、全frame/全コンテナ/全隙間ノードを構造で機械走査し、非auto-layout(layoutMode=NONE手動配置)・無記名Spacerノード・装飾でないlayoutPositioning=ABSOLUTE・abs座標やダミーノードで作った間隔を抽出して§0のabs:宣言と1:1突合する。違反0行でgreen（詳細は S7「auto-layout厳守逆監査」）。
- **9. 正直な報告** — 何を作ったか・どのコンポーネント/変数を使ったか・カバレッジ・未承認で保留したもの・残課題を明示。

## チーム編成
**レジェンド**: 工程＝アラビア数字 0→9（標準オーダー）、ロール＝丸数字 ①〜⑦。各ロールに対応工程を `=オーダーN` で併記。

- **マネージャー（自分・委譲不可）= オーダー0/3/4/8/9**: 規模判定／要件確定／設計ゲート／エスカレーション起票／全件カバレッジ最終確認（逆差分監査・OOUI逆監査・ページ編成逆監査含む）／完了可否。
- **① 偵察（Recon／読み取り）= オーダー1**: 既存類似画面の発見と構造把握＋既存コピーの横断走査。
- **② 棚卸し（Inventory／読み取り）= オーダー2**: 既存コンポ・変数・テキストスタイルの積極列挙。
- **③ プランナー（設計）= オーダー3**: ①②→画面構造＋マッピング表＋エスカレーション項目。
- **④ 実装（Implementer／書き込み）= オーダー5**: use_figma 実行。段階／領域分割。
- **⑤ 配線（Wirer／書き込み）= オーダー6**: §6専用段の実行者（④兼可、ただし段は別）。
- **⑥ レビュー（複数・敵対的／読み取り・全件被覆）= オーダー7**: 実装者と別。要素単位で割り合計で全件カバー。OOUI逆監査(f)の判定者も実装者と別個体。
- **⑦ シンセサイザ／反省（必要時）**: ファンアウト所見の統合。

並列OK=①②⑥（読み取り）。直列/分割必須=④⑤（書き込み衝突回避）。

## 詳細・判断基準

### 規模スケーリング（**最初に判定して編成と往復回数を決める**）
要件確定の最初にマネージャーが**規模クラス**を判定し、編成と承認往復回数を宣言する。

| クラス | 条件 | 編成 | 承認往復 |
|---|---|---|---|
| **XS-DS整** | 1画面・要素20未満・DS整備済 | ①②⑥兼ね、④単一段 | 1往復 |
| **XS-DS無** | 1画面・要素20未満・**DS未定義**（ファイル空） | マネージャー単独＋④単一委譲 | **1往復・短縮ルート発動**（後述「最小要件 × DS不在ルート」） |
| **S** | 1-2画面・要素50未満 | 標準編成（並列①②⑥） | 1往復 |
| **M** | 3-5画面、または要素100未満 | 標準編成、④領域分割 | 1-2往復 |
| **L** | 6画面以上、要素100以上、**または「操作可能なデータを扱う」画面**（ユーザーがソート/フィルタ/選択/編集で能動的にデータ集合を操作する画面） | 標準編成、④領域別並列 | **領域別2-4往復**（フィルタ仕様／ソート仕様／状態網羅／権限差 等を個別領域で詰める） |
| **XL** | 0→1ファイル＋複数画面＋DSも含めて構築 | DS不在順序（後述）必須 | 領域別＋段階別承認 |

**L判定の注意**: 「一覧」「ダッシュボード」「管理」等の語彙だけでL自動昇格しない。**ユーザーが能動的にデータ集合を操作する**（ソート列を押す／フィルタを変える／行を選択して一括処理する／編集する）画面のみL。静的なリスト表示やウィジェット並べは S/M で扱う。

**往復ルールの統一**: 「束ねる」と「分ける」の二段運用は規模クラス表が決定的。**領域内では束ねるが領域間では分ける**。XS〜Sは束ね1往復、Mは束ね1-2、Lは領域別2-4、XLは段階別。1往復厳守と領域別2-4は矛盾せず、規模クラスで分かれる。

### 要件理解の手順（0 の中身）
着手前に以下を**1枚に書き出してユーザーに見せる**。**質問は規模クラスに応じて束ね**、未指定項目は最初のQ&Aに必ず含める。

#### 必須項目（全規模共通）
- **画面の役割（1行）**: この画面で何が達成されるか。
- **画面サイズ・プラットフォーム・モード**: mobile（幅px）／web／responsive、light/dark対応の有無。
- **オブジェクトモデル＋OOUIビューインベントリ（表A/表B）**: 作成フレームリストは**先にビューを確定してから導出する**。動詞・手続き・種別は表Aに畳み、独立objectにしない（後述「OOUIビューインベントリ」）。
- **ページIA宣言（表C frame帰属／表D Page構成）**: フレームをどのPageに配置するかを**オブジェクト単位**で宣言（後述「ページIA宣言」「ページ編成ガイド」）。新規Pageは新しいオブジェクトが登場するときだけ。
- **作成フレームリスト（新規）**: 今回新しく作る画面/フレームを全列挙（表Bから導出）。**状態バリエの有無**を§0段で宣言（「通常状態のみ」／「状態複数あり：詳細は§4で承認」）。複数ありなら§3マッピング表で全状態フレームを列挙し、§4で一括承認。
- **改修対象リスト（既存）**: 既存画面に追加・修正を入れるフレームを全列挙。改修分は `figma-ds-edit` に委譲。
- **派生・補助画面の見込み**: 入力→確認→完了 等の派生画面を仮置きし、要件時にユーザーに「この派生は要るか」を4択で問う。
- **遷移配線リスト**: from(画面/要素) → to(画面) のペアを全列挙。**戻る・キャンセル動線**も必ず明記。**1画面案件で配線が無い場合は「配線対象なし」と明示**。**画面間遷移は§0で全列挙、画面内状態間遷移（フィルタ結果0件→クリア→通常等）は§3マッピング表確定時に追加**（§0の段階で全状態×全遷移の積を埋めようとしない）。
- **表示項目／入力項目**: 既存呼称で。入力はラベル／任意・必須／選択肢／初期値まで。
- **使うべき既存パターン**: 既に存在する同種画面の参照（あれば node-id）。
- **曖昧分岐**: 自分で決められないところを箇条書きで列挙→ユーザーへ。

#### OOUIビューインベントリ（表A/表B・委譲不可・§4承認対象）
作成フレームリストはこの2表から導出する（先にビューを確定してからフレームを起こす）。

- 表A オブジェクト棚卸し（動詞・手続き・種別はここに書く＝独立objectにしない）:

  | object(名詞) | 定義1行 | 主要プロパティ・状態 | 主要動詞(=後でアクションchipになる) |
  |---|---|---|---|

- 表B ビュー割当（1frame=1行、view列はcollection/singleの二択）:

  | frame | object | view(collection\|single) | このビュー上のmodeless action chip(動詞) | 状態バリエ | 出典(req/existing:<nodeId>/ds) |
  |---|---|---|---|---|---|

**表B auto-layout既定（1行・全frame共通の既定値）**: 全frame及び主要コンテナはauto-layout(layoutMode=HORIZONTAL/VERTICAL)で組み、余白・間隔はgap(itemSpacing)とpaddingで制御する。Spacerノード・非auto-layout手動配置・layoutPositioning=ABSOLUTEは既定で禁止(default-deny)。既定から外す要素だけを、表Bの『出典』列（または§3マッピング表の該当要素行）に逃がし弁タグ `abs:<code>` を1つ付けて明示列挙し（後述「auto-layout厳守の逃がし弁」）、各 `abs:` に『なぜgap/paddingでは表現できないか』の理由1行を必須化（無理由は差し戻し）。タグの無い要素は既定=auto-layout強制。列挙した `abs:` 例外は§4の一括承認に束ねる（出典タグ/proc:コードと同型で新しい宣言列は作らない）。

**宣言ルール**: ①全作成フレームは表Bに1行 ②view列はcollection/singleのみ ③collection/singleに収まらないframeは `proc:<code>` / `cross-object:` / `non-object:` タグ＋理由を書き§4で個別承認（default-deny、無宣言フレームは§8(f)で削除） ④動詞・種別・手続きはframe化せずA/Bに畳む。

#### ページIA宣言（表C/表D・§4承認対象）
OOUIビューインベントリ表Bと1:1で連動させ、2表を出す:

- 表C frame帰属表:

  | frame | object | view(collection\|single\|sub) | state | 配置Page名(番号prefix付き) |
  |---|---|---|---|---|

- 表D Page構成表:

  | Page名(番号prefix) | 含むオブジェクト | frame数 | 外向き配線(→他Page frame) | 分割理由(あれば) |
  |---|---|---|---|---|

**宣言ルール**: ①1オブジェクト→原則1Page。分割する場合はプレフィックス共有(例「メンバー／一覧」「メンバー／詳細」)＋理由を明記 ②各Pageは最低2frame(孤立Page禁止。横断Overview等の例外は明示) ③ストーリー由来のPageは作らず配線で代替、通し閲覧が要るなら既存frameを複製せず参照を並べる薄い『フロー俯瞰』Page1枚に限定 ④ビューインベントリ表Bの各行がどのPageのどのframeに落ちるかを1:1で対応付け、新規Pageは宣言した新オブジェクトのみ許可。ユーザー承認なしに実装着手しない。

#### データ操作系画面（L規模）の追加項目
ユーザーが「一覧」「管理」「ダッシュボード」「テーブル」等の語を含む依頼をしたら、以下を**領域別Q&Aで追加確認**:
- **データソース**: 何のデータ／レコード規模／ロード方式（一括 / 無限スクロール / ページネーション / カーソル）
- **ソート仕様**: 対象列／単一or複数キー／URL保持の要否
- **フィルタ仕様**: 単一/マルチ選択、AND/OR、即時/適用ボタン、URL保持、保存ビュー要否
- **状態網羅レベル**: 通常のみ／+空+エラー／+全網羅（後述「状態バリエの分類別ホワイトリスト」）
- **ロール分岐**: admin/user/読み取り専用/担当者 等の表示差の有無
- **一括アクション**: 行選択・一括変更・削除等の要否
- **列カスタマイズ**: 表示列選択・並べ替え・幅調整の要否
- **エクスポート**: CSV／PDF 等の要否

#### 曖昧指示の扱い（「よくあるやつ」「いい感じ」等）
9項目テンプレを丸投げしない。2点セットで確認:
1. **最小コア版**: 既存DS/既存画面で確実に出典が取れる項目だけで組んだ最小構成（例: 商品詳細なら 画像／商品名／価格／主CTA）。
2. **拡張候補リスト**: 出典が無く判断が割れる項目を **5〜8件**で箇条書き。**各候補に出典タグ必須**: `existing:<画面>` / `ds:<componentId>` / `similar:<参考機能>` / `guess:`（出典なし・推測）のいずれか。**`guess:` が3件以上を占めるなら、その時点でスコープ自体を要件主に確認するメタ質問**（「この画面は何が達成できれば完了か」）に切り替える。

### 文言・項目の取り扱い（最重要・鉄則2）

#### 出典タグの分類（実流通7種）
| タグ | 意味 | 使ってよい場面 |
|---|---|---|
| `req:` | 要件本文に明記／動詞流用 | 要件文字列をそのまま画面に置くとき（要件「ログインできる」→ ボタン「ログイン」可） |
| `ds:<componentId>` | DSコンポ既定文言 | コンポ既定テキストをそのまま採用 |
| `existing:<nodeId>` | 既存画面の同役割文言 | 表記揺れ吸収のため既存表記を流用 |
| `inherent:` | 機能必須要素（後述ホワイトリスト） | 入力欄ラベル等の最小限の機能語 |
| `placeholder:` | 表示用ダミーデータ | サンプル氏名・金額・商品名等 |
| `guess:` | 出典なし・推測 | **拡張候補リスト内のみ可**。本実装の画面テキストに `guess:` を残すのは禁止 |
| `primitive-def:` | DS不在ルートで primitive 変数を初回定義する瞬間の生値 | 鉄則4の生値禁止の唯一の例外。設計マッピング表に明示（後述「DS不在ファイル」） |

（注: 文言の出典は実質 `req:/ds:/existing:/inherent:/placeholder:/guess:` の6分類で運用し、`primitive-def:` は変数定義の生値タグ。両者の和集合で実流通7種。`guess:` は本実装に残さない。）

#### 機能必須要素ホワイトリスト

##### 共通（フォーム・詳細・コンテンツ系）
- 入力フォームの送信ボタン・キャンセル/戻る。**ボタンラベルは (a) 要件文中の動詞を流用（"ログインできる" → "ログイン"）／ (b) 既存画面の同役割表記を流用**。発明禁止。
- 値変更系の「送信中／成功／失敗」状態の表示位置（**メッセージ文言**はエスカレ対象）
- バリデーションエラーの表示領域（メッセージ文言はエスカレ対象）
- ヘッダタイトル・戻る矢印
- 入力欄ラベル（最小限の機能語：要件「メール」→ 「メールアドレス」可、揺れ吸収はユーザー確認）
- input type に伴う技術的属性（password/email 等）

##### データ操作系（一覧・テーブル・管理画面）
- **ソート**: 列ヘッダの▲▼アイコン、ソート状態の視覚表現
- **フィルタ**: 絞り込みトリガ、フィルタpill、適用、リセット、クリア、条件追加
- **ページネーション**: 前へ/次へ、件数表示、ページサイズ選択
- **選択**: 全選択チェックボックス、N件選択中バー、選択解除
- **エラー状態**: 再試行ボタン、エラーメッセージ表示領域（文言はエスカレ）
- **空状態**: empty data vs empty filter の使い分け表示領域（文言はエスカレ）
- **ローディング**: スケルトン / スピナーの配置位置

**ホワイトリスト外**（確認用再入力、現在値表示、ヘルプ文、注意書き、補足説明、空状態メッセージ文言、業務フロー固有の項目）は引き続きエスカレーション対象。**迷ったらエスカレ側に倒す**。

#### 表示vs文言の境界
- 「表示位置を置く」と「**そこに置くテキスト**」を分離。位置・領域確保は `inherent:` で許可、メッセージ文言は別途エスカレ。
- `inherent:` で確保したフレーム/領域は§0「作成フレームリスト」の状態バリエ宣言と一致させる（暗黙に新規フレーム/状態が生えるのを防ぐ）。

#### `inherent:` と `placeholder:` の使い分け
- `inherent:` は **UIの機能語**（ラベル「メールアドレス」、ボタン「ログイン」等）。
- `placeholder:` は **データの仮値**（商品名「Tシャツ サンプル」、金額「¥3,980」等）。
- `placeholder:` を UIラベル・見出し・案内文に使うのは禁止。
- 「現在のメールアドレス: 〈登録メール〉」のような**表示**は、(a) その表示要素を出すか自体がエスカレ対象、(b) 出すと決まった後の値は `placeholder:`、(c) ラベル「現在のメールアドレス」は `inherent:` or エスカレ。

#### プロジェクト辞書化
データ操作系画面で初めて作るとき、ホワイトリスト外の機能語（「絞り込み」vs「フィルタ」、「適用」vs「決定」等）の**確定表記をプロジェクト辞書として記録**（DSページにテキストフレーム or コメントで）。次以降の画面はこれを参照し、毎回確認を繰り返さない。

### 要素・フィールドの出典とアンチブリード（鉄則2の「要素版」・最重要）

鉄則2は**文言**の出典を縛る。同じ規律を**要素・フィールドそのもの**（入力欄／カード／
セクション／ボタン等、「これが画面に存在するか否か」）にも適用する。文言が正しくても、
要件にないフィールドが丸ごと生えるのを防ぐため。

- **全要素は出典必須（default-deny）。** 各フィールド/要素は **(a)要件 `req:` /
  (b)このサービスの参照画面 `existing:<nodeId>` / (c)DS既定・機能必須ホワイトリスト
  `inherent:`** のどれかに辿れること。辿れない要素は**置かない**。「フォームは指定された
  フィールドが全てで、それ以外は無い」を既定にする（「普通こうだから」で足さない）。
- **アンチブリード（他案件の混入禁止）。** 記憶・文脈・別プロジェクトに出てくる要素を
  この画面に混ぜない。**典型事故**: 古着サービスの「店主に呼んでほしい名前」を無関係な
  会員登録フォームに足す類。出典になるのは **“今回の要件”** と **“このサービスの参照画面”**
  だけ。ロードされた他案件の記憶は背景情報であって出典ではない。同種画面を参照するときも
  「別サービスの同種画面」からフィールドを写すのは `guess:` 扱い（=本実装不可、提案へ）。
- **「良かれ」は行動でなく提案へ。** 入れた方が良いと判断した要素は、勝手に置かず
  §0/§4の「拡張候補リスト（`guess:`）」に積んでユーザー承認を取る。承認なき `guess:` 要素を
  本実装に置くのは禁止（コピーの `guess:` 禁止と同じ扱い）。
- **§3マッピング表に要素行の出典タグを必須化。** 各要素行に上記タグを書く。タグの無い
  行は設計を通さない。
- **§8で逆差分監査（実装→宣言）。** §8の件数突合に加え、**実装に存在する全要素を列挙し、
  各々が§0作成リスト/§3マッピング表に宣言済みか**を逆向きに突合する。宣言に無い要素が
  実装に出ていたら**ブリード/スコープ膨張として削除、必要ならエスカレーション**。
  （従来の「設計N≦実装N」は template 繰り返しのみ許容。それ以外の未宣言要素は不可。）

### 手続き的フロー逃がし弁（proc:理由コード・default-deny・根拠宣言必須・閉じた列挙）
collection/single で書けない frame は閉じた理由コードを1つ付ける（§3エスカレ⑥／§4承認対象）:

- `proc:auth` — 認証/資格情報handshake(閲覧可能なドメインobjectが無い)
- `proc:txn-commit` — 取引の原子性＋ドメイン必須の順序(決済・不可逆submit。順序がUI都合でなくドメイン制約)
- `proc:dependent-branch` — 後続フィールド集合が前の回答で本質的に分岐(フラット提示だと無効な組合せを問う)。単なる値選択(単発/定期・ロール・種別)は分岐でなくプロパティ→不可
- `proc:legal-consent` — 外部規定の強制熟慮(同意/KYC/破壊的確認)
- `proc:onboarding` — 初回限定の教育的逐次(スキップ可・定常手段でない。同じ事をモードレスに行えるobjectビュー経路が別に存在することが条件)
- `proc:utility` — 非object面(404/error/splash/印刷ダイアログ)
- `cross-object:` — 複数objectを束ねるホーム/ダッシュボード/サマリ(各objectのcollection/singleへ確実に降りられること)

列挙外は `proc:other` のみ可、自動承認されず必ずユーザーへエスカレ(enum拡張は人間統制下)。各 `proc:` に1行の反証可能な正当化必須（「なぜobjectのcollection/single＋モードレスchipにできないか」を具体的ドメイン制約=順序強制/原子性/法定熟慮/真の分岐 で）。**無効理由(=書いたら差し戻し)**: 「綺麗」「ユーザーが期待」「作成フローだから」。例外スコープはcommit/handshakeの最小逐次スパンのみ。それが包むobject(Cart/Order/Account)の閲覧・編集ビューまで例外にしない。§8(f)で `proc:` 全件のリフレーム反証（レビュアーが能動的にOOUI化を試み、候補object名指し＋失敗するドメイン制約を文書化。書けなければproc:却下→OOUI化）を回す。

### auto-layout厳守の逃がし弁（abs:理由コード・default-deny・根拠宣言必須・閉じた列挙）
auto-layout/gap/paddingで組めない（=非auto-layout・絶対配置・隙間専用空ノードが正当な）要素は閉じた理由コードを1つ付ける（§0表Bの出典列 or §3マッピング表に明示／§4承認対象／§8(h)で全件反証）。出典タグ・proc:理由コードと完全同型:

- `abs:overlay-decor` — overlayバッジ/フォーカスリング/通知ドット等、**親のauto-layout計算に影響を与えてはいけない装飾**(layoutPositioning=ABSOLUTE。figma-component-design 鉄則6『絶対配置を使ってよい唯一のケース』と整合)。
- `abs:fixed-overlay` — モーダル/トースト/FAB/ドロワー等、画面フローから外して固定オーバーレイする最上位レイヤ。
- `abs:divider` — 意図的な区切り線(ただしfill/strokeのpaintを持つ実体=Spacerでない。隙間でなく『線』であることが条件。場当たり線で取り繕うのは別途禁止)。
- `abs:empty-state-slot` — 空状態/プレースホルダの予約枠(instance実体or名前付きslotで、将来コンテンツが入る宣言付き。無記名の純空ノードは不可)。
- `abs:test-canvas` — DSページ/別Pageに隔離した全property組合せ等の検証用テスト並べ(成果物Pageには置かない。隔離規律と整合)。

列挙外は `abs:other` のみ可、自動承認されず必ずユーザーへエスカレ(enum拡張は人間統制下)。逃がし弁は『なぜgap/padding/Boolean Visibilityで表現できないか』を装飾性・最上位レイヤ性・線の意味・空状態の予約という具体根拠で書けたときのみ通る。**無効理由(=書いたら差し戻し)**: 「綺麗」「位置合わせが楽」「なんとなく」「AL組むのが面倒」「普通こうする」「見た目が同じだから」。§8(h)で全 `abs:` のリフレーム反証（レビュアーが能動的にgap/padding化を試み、失敗する構造制約を文書化。書けなければabs:却下→auto-layout化）を回す。

### DS流用判定（State A/B/B'/C）
- **State A（完全一致採用）**: 既存使用箇所の用途・データ種・並びが新規用途と一致。そのまま採用。
- **State B（部分流用・要承認）**: 命名は近いが用途差あり。**「採用案 + 差分」をエスカレ票に切り出し**承認を得る。
- **State B'（合成内部欠落・要承認）**: 合成コンポ（Table/Form/Modal等）の本体は State A だが、内部の特定パターン（ソート可ヘッダ、行選択、フィルタpill等）が既存に無い。**欠落部品リストを1つのエスカレ票に束ねて**承認を取る。**束ねる上限は5項目まで**（超えるなら本体採用自体が不適切でState Cへ降格、または合成パターンの再設計をユーザーへ）。
  - **発見契機**: §2棚卸し時に「**合成コンポの内部探索**」を必須化（mainComponent内のバリアント軸とスロット候補を1段ドリルダウンして欠落を検出）。§5実装中に発覚した場合は**即§4へ巻き戻し**、束ね承認を取り直す。実装段で勝手にState Aで埋めず保留する。
- **State C（不採用）**: 命名一致でも用途違い・データ型違い。新規が必要なら鉄則5へ。

### 変数・スタイル必須ルール（鉄則4）

#### 色
- primitive/パレットを直接当てない。**役割名のセマンティック変数**を当てる。
- 役割: 主文字／補助文字／リンク・強調／枠・区切り／面（背景/カード/入力）／ブランド主色＋上の文字／状態色（成功/警告/エラー/情報）。
- モード（light/dark）対応。fill は必ず変数バインド。

#### タイポ
- **名前付きテキストスタイル優先**。役割／サイズに合えば適用。
- スタイルが無くタイポ変数の文化なら**変数バインド**。font-size と line-height は同じスケール段から必ずペアで。
- 任意px禁止。必要な段が無ければ「不足」を報告。

#### 間隔・角丸・影
- gap/padding/border-radius/elevation も生値禁止。space/radius/effect 変数があれば必ずバインド。

#### コンポーネント絡み
- インスタンス内部の色/サイズは**生値で上書きしない**。バリアント／プロパティで変える。
- 実データが入る入力欄は Filled 系バリアントを使う。

### 新規バリアブル/スタイル/コンポ作成時のエスカレーション（鉄則5）

エスカレーション票:
- **不在の証拠**: 全Nページ走査済み／検索した変数名・スタイル名・コンポ名／棚卸し結果のスクショ。**ファイル空・DS未着手のときは「ファイル空・DS未定義」の一行で代替してよい**。
- **流用案（あれば併記）**: 既存の近いトークン／スタイル／コンポを「これで代替できるか？」と並記。
- **新規提案**: 命名／カテゴリ・所在コレクション／モード（light/dark）／スコープ／初期値／参照元の使用箇所。
- **影響範囲**: 既存への波及（共有資産か、ローカルか）。
- **可逆性**: 後で消せるか／改名コスト。

承認後にだけ作成。**未承認の時は当該箇所を未完成として報告**（暫定生値で「とりあえず置いた」をしない）。

### DS不在ファイル（0→1）の作業順序
DSが整っていない/まだ無いファイルで新規デザインを起こす場合、**順序を飛ばさない**。各段でユーザー承認を取るが、**最小要件時は1往復一括承認に短縮**。

1. **primitive 変数**: color パレット／spacing／radius／typography スケール／effect。**生値はここで一度だけ定義**（鉄則4の生値禁止の唯一の例外。設計マッピング表に `primitive-def:` タグで明示）。
2. **semantic 変数 / テキストスタイル**: 役割名を primitive のエイリアスとして定義。
3. **トークンエイリアス**: state-success, link-default 等の追加層。
4. **atom/molecule コンポーネント**: Button / Input / Tag / Card / Avatar 等。
5. **画面組み立て**: 上記4までを使って画面を組む。

#### 最小要件 × DS不在ルート（XS規模の短縮ルート）
**判定条件**: ① 作成フレームが1画面 ② 要件記述が3文以下 or 入力項目3個以下 ③ 棚卸しで「DS未定義」確定。

このときに限り:
0. **§1現物確認・§2棚卸しは「空でした／DS未定義」の1行宣言で完了**（儀礼スクショ不要）。
1. **要件補完を1往復にまとめる**: 画面サイズ／プラットフォーム／submit後の遷移／エラー時の文言扱い／ブランド色の有無／**このファイルにDSや既存画面が無いがゼロから作ってよいか（別ライブラリ参照の有無）**を1往復のQ&Aで一括確認。
2. **DS最小セットを束ねて1回で承認**: ユーザーに対して**「ゼロ案＋拡張提案」の2列**で提示し、yes/no/採用範囲を選ばせる。プリセットを押し付けない（鉄則2準拠）。
   - **ゼロ案**: 「今回の要件に直接必要な分のみ」（例: ログイン画面なら neutral 黒/白/中グレー1段／space 4の倍数3段／typo 2段／radius 1段）
   - **拡張候補**: 「将来再利用するなら」neutral 5段までの拡張／brand色／danger等の追加 — **各候補に理由を付け**、ユーザーが ◯×で個別選択できる形に。
   - **暗黙のセマンティック多重化（surface/border/danger 等）は提案に含めるが、ゼロ案の時点では必須にしない**。ユーザーが要らないと言えば作らない。
3. **atom 最小セット（Button / TextField）のみ事前承認**で作成、それ以外は画面段で発覚した時のみ追加。
4. **段階実装は領域単位ではなく画面単位で1回**でよい。

**短縮ルートでの語確定優先順位**: プロジェクト辞書がまだ無い状態でラベル/ボタン文言を確定するとき、(1) 要件本文の動詞流用（`req:`）→ (2) `inherent:` ホワイトリストの機能語（既存DSや既存画面が無い場合は、共通辞書の標準語: 「ログイン」「メールアドレス」「パスワード」「保存」等）→ (3) 要件補完Q&Aへ追加、の順で決める。**1往復のQ&Aを破ってまで個別の語をエスカレしない**（共通辞書の標準語で進めて最終報告で明示）。

短縮ルートを使ったことは、最終報告で必ず明示する。

### 設計マッピング表のフォーマット（③で必須）

要素マッピング表（frame行に object / view / modeless-action 列を追記）:
```
| frame | object | view | section | element | cardinality | scope | component(id/variant) | text | modeless-action | 出典 | binding(color/typo) |
|---|---|---|---|---|---|---|---|---|---|---|---|
| 商品詳細 | 商品 | single | ヘッダ | 戻る | single | 新規 | IconButton/back | — | — | ds:<id> | color/icon-default |
| 商品詳細 | 商品 | single | 主情報 | 商品名 | single | 新規 | Text | サンプル商品名 | 編集chip | placeholder | typo/heading-m |
```

- `object` / `view(collection|single|proc:<code>)` / `modeless-action`: frame=オブジェクトのビューであることを表現（鉄則13・OOUIビューインベントリ表Bと整合）。`collection/single` に収まらない frame は `proc:<code>` / `cross-object:` / `non-object:` を入れエスカレ⑥（→「手続き的フロー逃がし弁」）。
- `cardinality`: `single` / `template(N=8)` / `dynamic` のいずれか。
- `scope`: `新規` / `改修` のいずれか。改修行は混在運用時に `figma-ds-edit` へ委譲する対象（reaction張りも改修扱い）。
- 実装マッピング表で許容される出典は、宣言で使う実流通7種のうち `guess:` を除いた **6種**（`req:` / `ds:` / `existing:` / `inherent:` / `placeholder:` / `primitive-def:`）。このいずれかで埋まらない行はエスカレ対象。`guess:` は拡張候補リスト専用で、実装マッピング表に残すの禁止。

#### N突合の定義（cardinality対応）
- **マッピング表のN（設計N）**: `single` は1、`template(N=K)` は1（テンプレ1行）、`dynamic` は1（型のみ記載）。**設計承認はテンプレ1行で完了**。
- **実装後の要素件数（実装N）**: `single` は1、`template(N=K)` は K、`dynamic` は実体ベース。
- **レビュー件数（レビューN）**: `single` は1全件、`template(N=K)` は**テンプレ構造正しさ＋K個全体の placeholder 多様性チェック**（テンプレ通りでない異常がないかを全件目視。同じテンプレ文字列でOK）、`dynamic` は K 個サンプリングではなく実体ベース全件。
- **件数突合**: 設計N≦実装N＝レビューN。「設計1：実装K」は許容、「実装K：レビューM(<K)」は禁止。

### 状態バリエの分類別ホワイトリスト
§0「作成フレームリスト」で「状態複数あり」を選んだとき、§3で全状態を列挙する際の**最低網羅参照**。**種別/定期/立場はframe追加でなく、同一(object,view)行の状態/プロパティとして扱う**（種別選択や定期/単発の分岐をframe化しない＝鉄則13）。

#### フォーム画面
通常／空／検証エラー／送信中／完了／不可

#### データ表示画面（一覧・テーブル・ダッシュボード）
通常／**完全な空（データ0件）**／**フィルタ結果0件**／**検索結果0件**／**初回ロード**／**部分ロード**／**取得エラー（再試行可）**／**権限不足**／**選択中**

#### 詳細画面
通常／読込／取得エラー／削除済／**競合（他者編集中）**／**ロール別表示差**

§0段ではこれら全部を列挙せず**「状態複数あり」の有無**だけを宣言。複数ありなら§3で全状態フレームを列挙し、§4で一括承認。§0で全状態を埋めようとして質問数を膨らませない。

### ページ編成ガイド（オブジェクト単位＝既定ページ単位）

■ 結論: 「ストーリーごと」でも素の「機能ごと」でもなく、**既定はオブジェクト単位**（1オブジェクト=1Page）。多くの機能はオブジェクトと一致するので“機能ごと”に近いが、判断軸は必ず『どのオブジェクトのPageか』に置く。ストーリー(複数オブジェクトを貫く業務フロー)は**Pageの単位にしない**——それをPageにすると同じオブジェクトのビューが複数フローPageに重複・散在し、まさに今回の事故とある実プロジェクトでのフロー退行（過去の教訓）を物理レイアウトに固定化する。フローはプロトタイプ配線(必要なら俯瞰1枚)で表す。

■ 迷わない決定ルール:
- **STEP0 オブジェクト同定**: 作る画面群が扱うメインオブジェクトを列挙(例 メンバー/面談/目標)。各frameを『どのオブジェクトの・どのビュー(collection|single|サブビュー)・どの状態』に必ず分類。分類できないframeは作らない(OOUI出典ゲートと同型のdefault-deny)。
- **STEP1 既定割当**: 1オブジェクト=1Page。collection＋single＋全状態＋そのオブジェクト固有のモーダル/インライン操作結果を同一Pageに並べる。
- **STEP2 散在判定**: あるオブジェクトのビューが2つ以上のPageに跨ったら違反(=1画面1ページのスメル)。1frameだけのPageを作らない。
- **STEP3 ストーリー**: 複数オブジェクトをまたぐ業務フローはPageを切る理由にしない→配線で表現、俯瞰は横断1枚『フロー俯瞰』に縮小サムネ/矢印で。フローごとにPageを増やさない。
- **STEP4 分割閾値(同一オブジェクト内のみ)**: 1Pageが破綻するときだけ分割可(目安: そのオブジェクトのframe数が概ね24枚超、または一覧と詳細で関心が完全分離)。分割時もプレフィックス共有＋隣接配置。立場/種別/日程/定期/リマインドは新オブジェクト扱いせずプロパティ・状態・アクションとして同一Pageに収める。
- **STEP5 優先順位**: オブジェクト軸＞ストーリー軸。複数オブジェクト合成画面(ダッシュボード等)は主オブジェクトで帰属、主が無ければ『横断/Overview』Pageへ(`cross-object:`)。
- **STEP6 既存ルール整合**: node-id指定はin-place、『別ページ』は新規Page、該当無し/複数候補で割れるなら着手前にユーザー確認。

■ わかりやすい体裁(まとめ):
- **Page命名**: 番号prefix＋オブジェクト名で固定(例 `00 Cover/IA索引`, `10 メンバー`, `20 面談`, `30 目標`, `90 Overview/ホーム`, `99 Archive`)。10刻みで挿入余地。Pageの並び自体がオブジェクトモデルの並びになる。分割は全角スラッシュで従属明示(`メンバー／一覧`)。
- **1Page内のframe並び**: 左→右に view種別(コレクション→シングル→サブビュー/モーダル)、上→下に状態(default→empty→loading→error→権限/立場差)。左右関係＝OOUIのcollection⇄single往復をキャンバス上でも表現。
- **セクション区切り**: Figmaの Section 機能(Group/外付けTEXT見出しの偽装は不可)で『コレクション』『シングル』『状態バリエ』『アクション派生』に分け、どのオブジェクトPageも同じSection骨格を使う。
- **frame命名**: `{オブジェクト}/{ビュー種別}/{状態}`(例 `メンバー/コレクション/default`, `面談/シングル/編集中`)で機械突合可能にし、view inventory各行と1:1一致。命名を見れば散在が即検出できる。
- 各Page左上に小さなIAヘッダ(担当オブジェクト名/内包ビュー一覧/外向き遷移)を置きPageを自己文書化。
- **隔離**: DS/共有コンポ本体はDSページ、全property組合せ等の検証用テスト並べframeは別Page/別領域へ隔離し成果物Pageに混在させない。
- **散在(違反)と横断(正常)の機械的区別**: 同一オブジェクトのビューが割れている=散在/違反、別オブジェクトのビューへ繋がっている=横断/正常。

### 段階的作成（暴走防止）
- 1画面を1回の `use_figma` で完成させない。論理ブロック単位（コンテナ／ヘッダ／フォーム1ブロック／ボタン領域／配線）で分割。
- 各段の終わりに**自分でスクショ確認**→OKなら次。
- `use_figma` 各呼出後にレスポンス検証（エラー無し、返却ID整合）。失敗時は**即停止 → 現状 `get_screenshot` → 原因特定 → 修正案提示**。連投禁止。
- 同一ファイル同時書き込み禁止。並列実装するなら重ならない領域に分ける。

### プロトタイプ配線（§6）
新規/改修問わず、画面間遷移は**必ず Reaction (On click → Navigate to → 対象フレーム)** で配線する。FRAMEの命名や見た目だけで「繋がっているつもり」にしない。

- **専用の一段**として切る。骨格・テキスト確定後に reaction だけ流す。
- **配線対象は §0「遷移配線リスト」と1:1突合**。リスト外の遷移を勝手に張らない。
- **戻る・キャンセル**の reaction も必ず張る。各遷移行は **trigger と back導線をペアで宣言**（例: 「設定→通知設定」と「通知設定→設定（戻る）」をペアで列挙、片方だけのリストは§0で差し戻し）。
- 共通コンポ（MyMenuRow等のインスタンス）の reaction は**インスタンス上で個別配線**するのが基本（main側に張ると全画面波及する事故）。
- **配線件数のN突合**: 遷移配線リスト件数 = 実配線件数 = レビュー件数（戻る/キャンセルも1件として数える）。
- **状態保持遷移**（一覧→詳細→戻る時にフィルタ・ソート・ページ位置を保持したい）の場合: (a) フィルタ済み状態の一覧フレームを別途用意して双方向reactionを張る、または (b) `Navigate(preserve:filter,sort,scroll)` を遷移配線リストに記載し実装側に注記、のいずれか。**「戻る=初期一覧」にしてはいけない**ケースは要件確定で必ず確認。
- **1画面案件で遷移配線リストが空のとき**は本段を「対象なし」で完了宣言してよい。
- **配線後に OOUI逆監査(f) の reactionグラフ部を再実行**（配線前は判定不能。collection⇄single の往復到達と線形ウィザードエッジの不在をここで裏取りする）。

### 全件レビュー（⑦⑧）
- 配置した全要素＋全配線を**列挙→全部**を検証対象に（代表サンプル禁止）。広ければレビューをファンアウトし合計で全件カバー。
- 各要素で以下を1件ずつ判定:
  - 見た目がスクショで意図どおり。
  - **instance 実体監査**: node type が `INSTANCE`、mainComponent が想定通り、detach されていない。
  - **バインド監査**: fill/stroke が変数バインド、テキストがテキストスタイル or タイポ変数バインド。生値の残骸が無い（DS不在時の `primitive-def:` 除く）。
  - **文言監査**: 6つの出典タグのいずれかが付いており、`guess:` や発明文が紛れていない。
  - **プロトタイプ配線監査**: §0「遷移配線リスト」と実配線が1:1で一致。trigger / action / destination を全件確認。戻る・キャンセルも含む。インスタンス上に張られているか（main側への誤配線がない）。
  - **改修対象のスコープ監査**: §0「改修対象リスト」外の既存frameに変更が入っていない。MyMenuRow等の追加行が detach されておらず instance のまま、項目順が壊れていない。
  - **OOUI/モードレス監査**: 各frameが「どのオブジェクトのどのビュー(collection/single)」として成立しているか・別オブジェクトの混入や動詞frame化(ウィザード化/step indicator/「次へ/戻る」主動線)でないか。
- **カバレッジ明示**: 「対象◯件／確認◯件」「配線◯件／確認◯件」。間引きは禁止。

### OOUI逆監査（§8(f) 詳細・実装→宣言）
(e)逆差分監査の直後に実行。判定者は実装者と別個体（＝§7敵対レビュー）。

1. `get_metadata` で実装済み全top-level frameを列挙。
2. 各frameを機械判定:
   - (i) 表Bに対応行があるか。無い=未宣言フレーム→ブリード/退行として削除orエスカレ。
   - (ii) view種別を構造で裏取り(tag自己申告で合格にしない): collection=同型反復(リスト/テーブル/カードグリッド)が画面の主役 / single=単一オブジェクト1件の属性・状態提示。
   - (iii) modeless退行シグナル検査(1つでも該当=§4へ巻き戻し): frame名/ヘッダが動詞+ステップ名(「○○を作成」「ステップ1/3」「方法を選ぶ」)、step indicator/進捗ステッパー(生rect+TEXTの偽装ステッパーも疑い構造で見る)、frame下部の「次へ＋戻る」ボタン対、全画面ラジオ分岐(種別/今すぐ/単発/定期)、next/back以外の操作が無く背後にビューが無い閉じ込め。
3. reaction一覧を有向グラフ化: (a)入次数1/出次数1の一本道チェーン (b)「次へ/戻る」trigger比率 (c)collectionへ戻るエッジの無い「完了」終端 (d)decision frameのfan-out を抽出。健全形(collection⇄single双方向＋overlay→元ビュー復帰)から外れたエッジを列挙。**命名がオブジェクト名詞でもグラフが線形ウィザードなら不適合(命名は偽装可能、構造が正本)。**
4. 各objectでcollectionとsingle双方が存在し相互到達可能か(往復reactionが各1行立つか)を突合。片方欠落/片方向のみ(選択対象が次ビューに継承されない含む)は欠陥。
5. §6配線前でreaction未取得なら「判定不能」(=適合ではない、空振り≠不在)。命名＋frame系列＋画面内mode固定UIの静的シグナルで暫定判定し、§6配線後に再監査。
- **出力**: 「frame N件中 写像済みN / 未写像N」「線形チェーンエッジN件」のカバレッジ形式・間引き禁止。違反0行でgreen。
- **proc:全件のリフレーム反証**: レビュアーが能動的にOOUI化を試み、候補object名指し＋失敗するドメイン制約を文書化。書けなければ proc:却下→OOUI化（→「手続き的フロー逃がし弁」）。

### ページ編成逆監査（§8(g) 詳細・実ファイル→宣言の双方向突合・間引き禁止）
0. 全Pageを実数取得で走査(`get_metadata`(nodeId無し)は1ページしか返らない罠を回避、空振りを不在と解さない)。各frameを実際の(object, view, state)に分類。
1. **散在検査**: 実ファイルから object→Page帰属集合を再構築し、|Page集合|>1 のオブジェクトを抽出。表Dで分割申告済み以外は違反→統合。
2. **孤立検査**: frame数=1のPageを全列挙。宣言で例外申告(横断Overview等)済み以外は違反。
3. **帰属/default-deny検査**: 実在するが宣言に無いPage/frame=スコープ膨張で弾く、宣言にあって不在=欠落で弾く(双方向)。
4. **横断検査**: 表Dの外向き配線の各to-frameが実在Pageに到達するか。別オブジェクトPageへの到達は正常(=横断)、同一オブジェクトのビュー割れは違反(=散在)。死配線/未配線を検出。
- **出力**: 違反frame/Page名のリスト。空であれば合格。差分が出たら配置を直すかマニフェストを更新して再監査。「散在していないはず」は不可、object→Page集合の実測でのみ合格。

### auto-layout厳守逆監査（§8(h) 詳細・実装→宣言・判定者は実装者と別個体＝§7敵対レビュー）
(g)ページ編成逆監査の直後に実行。§0表Bの『auto-layout既定＋abs:例外宣言』に対し、実装を構造で逆突合する（見た目でなく get_metadata の layoutMode/itemSpacing/padding/fills/strokes/children/layoutPositioning で判定）。

1. `get_metadata` で実装済み全frameと全コンテナ(子2つ以上を持つノード)を実数列挙（`get_metadata`(nodeId無し)は1ページしか返さない罠を回避、全Page走査）。
2. 各コンテナを機械判定（**検出シグナル2: 非auto-layoutコンテナ＝手動配置**）: 子を2つ以上持つコンテナで `layoutMode==NONE`(またはlayoutModeプロパティ不在)、子がx/y絶対座標で並び間隔がitemSpacing/paddingでなく座標差で作られているものを列挙→§0でabs:宣言が無ければ違反。単一子のルート/キャンバス直下frameは除外。
3. 全ノードからSpacer候補を抽出（**検出シグナル1: 隙間専用空ノード＝Spacer本体**）: **(a)fills空 かつ (b)strokes空 かつ (c)子ノード0 かつ (d)非TextNode かつ (e)width or heightの一方だけ実寸(他方は0/極小 or 親方向にFILL)** を全件列挙。塗りも線も文字も中身も無く間隔だけ作る純粋な隙間ノード。誤検出ガード(後述)で除外し、残った無記名隙間ノードは違反。
4. auto-layoutコンテナの不一致・濫用を抽出し§0のabs:宣言と1:1突合（宣言外は違反）:
   - **検出シグナル3**: itemSpacing≒0 かつ padding≒0 なのに子の間に視覚的ギャップ＝ギャップが子側のダミー空ノードや固定sizeの透明子で作られている（itemSpacing/paddingと実測ギャップの不一致で検出）。
   - **検出シグナル4: 先頭/末尾ダミーによる擬似padding**: paddingLeft/Right/Top/Bottomが0で、代わりに端に塗り無し・文字無しの空ノードを置いて内側余白を作っている。
   - **検出シグナル5: layoutPositioning==ABSOLUTEの濫用**: 装飾(overlayバッジ/フォーカスリング)でない実コンテンツ・フロー参加要素がABSOLUTEで配置され、本来gap/paddingで並べるべき要素を絶対座標で逃げている。装飾(abs:宣言済み)以外のABSOLUTEを違反とする。
   - **検出シグナル6: 手動x/yによる間隔調整**: auto-layout子であるべき要素のx/yが、AL詰め後の期待座標(itemSpacing積算位置)からずれている＝overrideで手動ずらしして間隔を足している（absoluteBoundingBoxの実座標とAL理論座標の差分で検出）。
5. **§0宣言との突合**: abs:宣言した例外が(i)実在し(ii)逃がし弁の閉じたコードに該当し(iii)理由1行が反証可能か(「綺麗」「位置合わせが楽」「なんとなく」「AL組むのが面倒」は無効→差し戻し)を確認。**宣言abs件数=実装ABSOLUTE/非AL件数を厳密N突合**。各abs:のリフレーム反証（レビュアーが能動的にgap/padding化を試み、失敗する構造制約を文書化。書けなければabs:却下→auto-layout化）を回す。
- **誤検出ガード（=違反にしないもの・機械的除外）**: ①空状態/プレースホルダ枠（空でもinstance実体・名前付きslot・将来コンテンツが入る宣言済み枠は正当。無記名の純空ノードのみSpacer判定） ②意図的divider（fill or strokeのpaintを持つRECTANGLE/LINEは区切り線であってSpacerでない＝塗り/線の有無が分岐点。ただし§構造系『場当たり線で取り繕わない』は別途適用） ③装飾overlay（abs:overlay-decor等で宣言済みのlayoutPositioning==ABSOLUTE） ④min/maxサイズ（コンテンツノードのminWidth/minHeightは隙間ノードでない）。これら4種は『塗り/文字/子/instance実体/宣言のいずれかを持つ』で機械的に除外し、純粋な無記名・無paint・無子の幅or高さだけノードのみをSpacer違反とする。
- **出力**: 『コンテナN件中 auto-layoutN/非ALN』『Spacer候補N件中 違反N/ガード除外N(各除外理由)』『abs:宣言N=実装例外N』のカバレッジ形式。代表サンプル禁止・間引き禁止・全件列挙。**違反0行で初めてgreen**。違反が1件でも出たら修正(gap/paddingへ移行・Spacerノード削除・layoutMode付与)、ドメイン上どうしても非ALが要るならabs:でエスカレーション。§6配線とは独立に実行可。

## やってはいけない
- 規模クラス判定をせずに走り出す（往復数とチーム編成が決まらない）。
- 要件未確定・曖昧分岐が残ったまま手を動かす。
- 9項目テンプレを丸投げして個別承認を連鎖させる（規模に応じて束ねる）。
- 「作成フレームリスト／改修対象リスト／遷移配線リスト／ビューインベントリ／ページIA」を宣言せずに走り出す／リスト外を勝手に作る。
- 雰囲気で見出し・サブコピー・補足文・プレースホルダ・空状態テキスト・ヘルプ文を足す。
- 「業界の典型はこう」で色・項目・要素を勝手に追加する。
- 拡張候補リストに出典タグなしで項目を並べる／`guess:` を実装マッピング表に残す。
- 生hex／raw font-size／raw line-height／raw spacing／raw radius を置く（DS不在時の primitive 初回定義のみ例外）。
- 棚卸し不徹底のまま「無い」と断じて新規変数／新規コンポを作る。
- 既存DS部品を命名一致だけで採用する（用途違い事故）。
- 合成パターン（Table等）の内部欠落を「State A完全一致」で素通りする。
- 共有資産への追加を、ユーザー承認なしで行う。
- DS に実体がある部品を FRAME+TEXT で手描きする（必ず instance）。
- **Spacerノード（無記名の無paint無子の幅/高さだけノード）を挟む／コンテナを layoutMode=NONE の手動配置で組む／間隔を abs座標・ダミーノードで作る**（=auto-layout厳守違反、§8(h)で弾く）。例外は §0 で `abs:<code>` 宣言＋理由1行を取ったものだけ。
- `use_figma` のエラーを見逃して次の操作に進む。
- **プロトタイプ配線を省略する**／戻る導線をペアで宣言しない／状態保持遷移を単純Navigateで素通りする。
- 共通インスタンスの main 側に reaction を張る（全画面波及事故）。
- 改修対象リスト外の既存frameに「ついで」で変更を入れる。
- レビューを代表サンプル数件で済ます／繰り返しテンプレを全件突合の数に含めない。
- 未承認の保留事項を報告で隠す／暫定対応で埋めたことを黙る。
- **動詞/工程フレーム・step indicator・「次へ/戻る」主動線・全画面分岐選択を作る**（オブジェクトのビューに起こさずタスク/ウィザードに退行する＝鉄則13違反）。
- **種別/立場/定期/リマインドをフレーム化する**（オブジェクトでなくプロパティ・状態・アクションに畳む）。
- **tagロンダリング**（ウィザードstepを「single・状態=作成中」と言い換えて逆監査を素通りさせる）。
- proc:逃がし弁を「綺麗」「ユーザーが期待」「作成フローだから」で正当化する（=書いたら差し戻し）。commit/handshakeの最小スパンを超えて、包むobjectの閲覧・編集ビューまで例外化する。
- **1画面=1ページで機能をばらまく（散在）／1frameだけの孤立Pageを連発する／ストーリー(業務フロー)単位でPageを切る**（フローは配線で表現、新規Pageは新オブジェクトのみ）。
- node-id指定があるのに新規Pageへ作る／「別ページ」依頼を同一Page内の新フレームで済ます。
