# 自作Figmaスキルの入口（公式とは別系統）

**TL;DR**: Figma作業の最初に必ず通る入口/ディスパッチャ。1問で「どれを目指すか」を判定し、自作スキルを **1つだけ** ロードする。あわせて、全自作スキルに効く **共通禁止事項の唯一の正典（SOT）** をここに集約する。出力は「どのスキルへ進むか」の確定＋共通原則の適用。

Figma作業はまずここで **「どれを目指すか」を判定して、自作スキルを1つだけロード**する。
自分は `/mikeneko-figma` だけ覚えればよく、4つを毎回見比べて選ぶ必要はない。

**公式プラグインのスキルは振り分け対象に含めない。**
`figma-use` / `figma-generate-design` / `figma-generate-library` / `figma-generate-diagram` /
`figma-create-new-file` / `figma-code-connect` / `figma-swiftui` などは、各自作スキルが
内部で必要なときに呼ぶ「下請け（プラグインAPI層）」であって、ここで"目指す対象"にはしない。

## 判定（1問で割れる）

| やること | ロードするスキル |
|---|---|
| **既存**のものに手を入れる（restyle / 変数・テキストスタイル適用 / 生frame→component置換 / レビュー） | **`figma-ds-edit`** |
| **新規**に **部品1個**を起こす／**既存部品のproperty軸・sizingを大改修**する（ボタン/入力/バッジ/チップ等。variant・boolean・swap軸やslotの設計、全組合せテスト） | **`figma-component-design`** |
| **新規**に **画面・フロー丸ごと**（要件から既存DSの上に起こす＋プロトタイプ配線まで） | **`figma-design-create`** |
| **既存**の画面/フローが **実ユーザーの目的を達成できるか検証**する（体験のE2E。盲目セルフプレイで歩かせ、設計は直さない） | **`figma-e2e-test`** |

判定したら、そのスキルの鉄則・標準オーダーに従って進める。

**`figma-design-create` へ振り分けたら、まず「ビューインベントリ（オブジェクト×collection/single）」と「ページIA（オブジェクト単位のPage）」を確定してから着手する。** フレームをいきなり起こさず、先にどのオブジェクトのどのビューを作るかと、どのPageに何を置くかを宣言する（詳細はdesign-create側の§0/§3/§8ゲート）。

## 迷ったときの切り分け（グレーゾーン）

- **既存frameの中身を直す** ＝ edit。 **新しいframe/画面を起こす** ＝ design-create。
- restyleしながら既存部品を別のDS部品に **差し替える** ＝ edit（差し替え先は既存DS部品をinstance化）。
- 既存部品の **property軸を大改修** する／ **新規に部品を起こす** ＝ component-design。
- design-createの途中で「**DSに無い部品が要る**」と判明 ＝ いったんcomponent-designへ寄り道して
  先に部品を設計してから画面作成に戻る（手描き偽装で代用しない）。
- 複数該当するなら **「対象の単位（既存全体 / 部品 / 画面）」が一番大きいもの** を主にして、
  足りない部品だけ寄り道する。
- **体験を検証する（実ユーザーが目的を達成できるか歩いて確かめる）** ＝ e2e-test。
  **構造/DSの正しさをレビューする（変数バインド・instance実体・トークン適合）** ＝ ds-edit。
  e2e-testはテスト専用で設計を直さない。直す段になったら ds-edit / design-create へハンドオフ。

## 共通の禁止事項（全スキル共通の正典・SOT）

**このセクションは全自作スキル（ds-edit / component-design / design-create / e2e-test）に効く共通禁止事項の唯一の正典（SOT）。** 各ワーカースキルはこの規範を全文再掲せず、自分のゲートで実際に検証する項目だけを《一行宣言＋この入口§への→ポインタ》で持つ。規範を更新するときはこの1箇所を直せば全ワーカーへ伝播する。

詳細手順は各スキルに置く。ここは全作業に効く原則と、過去の事故から来る厳守事項を、下の7カテゴリ（構造系／部品系／配置系／出典系／OOUI系／品質系／進め方系）で束ねる。

### 構造系（構造・レイアウト）
- **Spacer要素（余白埋め用の空frame/rectangle）は禁止。** 余白・間隔は全て
  auto-layout の gap（item spacing）と padding で制御する。固定の隙間ノードを
  挟んで見た目を合わせるのは不可。
- **この原則は努力目標でなく機械的ゲートである。** 守れているかは見た目でなく構造（get_metadataのlayoutMode/itemSpacing/padding/fills/strokes/children/layoutPositioning）で判定する。各ワーカースキルは自分のゲートで実検証する: 画面新規作成は design-create §8(h)『auto-layout厳守逆監査』（着手前=§0表Bでauto-layout既定を宣言→実装後に全コンテナ/全隙間ノードを機械走査して逆監査）、既存編集は ds-edit 鉄則＋全件レビュー⑤⑥のauto-layout観点、**コンポーネント内部の機械則本体（sizing/デッドスペース）は figma-component-design §オートレイアウト設計＞デッドスペース禁止（TextField事件・正本）＋鉄則6『内側はオートレイアウト』が正典**。本入口は禁止原則のSOT、各スキルは一行宣言＋ここへの→ポインタで自足検証する。
- **場当たりの線・区切りで体裁を取り繕わない。** 「右側に線を引いただけ」の
  ような根拠のないデザインは禁止。各要素の意味に沿った構図で組む。

### 部品系（部品・コンポーネント）
- **アイコンボタンを使うときはアイコンも必ず指定する。** アイコン未指定の
  プレースホルダのまま置かない。入れるアイコン（名前/ノード）を決めて実体をセットする。
- **部品はinstance実体で配置する（手描き偽装禁止）。** 生frame・外付けTEXTで
  部品に見せかけない。検証は各スキルの最終ゲートで行う（配置作業=ds-edit⑥／
  design-create§8のinstance実体監査でレイヤー種別/プロパティ/slot利用を構造確認、
  スクショ一致で合格にしない。部品を新規設計する場合はcomponent-designの全組合せ
  テスト＋既存使用箇所への影響レビューがこれに相当）。
- **マスターComp（メインコンポーネント）を編集して見た目を変えない。** 要素を消す・
  変種に合わせる時はインスタンス側で `visible=false`（行コンテナごと畳む）。
- **コンポーネントの強調・マーキングで fill/stroke を変えない。** 全インスタンスに
  伝播して事故になる。マーキングは外側のラッパーで囲んで行う。

### 配置系（配置・対象）
- **置き場所のページは判定根拠を持って決める。** 既存の同種フレームがあるページ、
  または対象機能に対応するページに置く。該当が無い・複数候補で割れる場合は着手前に
  ユーザーへ確認し、自己判断で空いている所に置かない（「別ページに作って」は
  新規Pageを作る）。どのページに置くかは
  下記「既定のページ単位＝オブジェクト」で“どのオブジェクトのPageか”として具体化する。
- **node-id 指定はそのノードを in-place で編集する（複製・横並び禁止）。** Page を
  指定されたらその Page 上に作る。
- **勝手に新規コンポ・要素を作らない。** 作る前に既存DSに有るかを必ず確認する
  （検索やページ一覧の空振りは「無い」の証拠ではない。存在だけでなくslot/プロパティの
  capabilityまで読む）。重複作成しない。

#### 既定のページ単位＝オブジェクト（散在禁止・宣言→逆監査）
- **複数フレームを起こすときは「オブジェクト単位」でPageにまとめる（=1オブジェクト1Page、実務上の“機能”とほぼ一致）。** そのオブジェクトの collection(一覧)・single(詳細)・全状態・モードレスなアクション派生を**同一Page内に並べる**。「ストーリー/フロー単位」ではPageを切らない（フローはプロトタイプ配線で表現）。
- **「1画面=1ページ」で機能をばらまかない（散在禁止）。** 同一オブジェクトのビューが2つ以上のPageに跨る・1frameだけのPageが連続する、は違反シグナル。新規Pageを作ってよいのは“新しいオブジェクト”が登場するときだけ（「別ページに作って」=新規Pageは不変、node-id指定はin-place不変）。
- どのオブジェクトのPageかを判定根拠にする。配置IAは着手前に宣言し、最終ゲートで object→Page集合の実測で逆監査する（宣言→逆監査。本ゲートの詳細はdesign-create側）。

### 出典系（要素の出典・スコープ膨張・文脈ブリード対策）

「良かれと思って足す」は本人にとって違反に見えないので、自制では止まらない。
**意図ベースの禁止ではなく、出典ベースのゲートで縛る。**

- **全要素は出典を持つ。出典の無い要素は置かない（default-deny）。** 各フィールド/
  要素/コピーは次のどれかに辿れること: **(a)ユーザーの要件 / (b)“このサービス”の
  参照画面（node-id） / (c)DSの既定**。どれにも紐付かない要素は禁止。「フォームは
  指定されたフィールドが全てで、それ以外は無い」を既定にする（「普通こうだから足す」
  をしない）。
- **他案件の情報は出典にならない（アンチブリード）。** 記憶・文脈・別プロジェクトに
  出てくる要素を混ぜない（例: 古着サービスの「店主に呼んでほしい名前」を無関係な
  会員登録に足す、等）。出典になるのは“今回の要件”と“このサービスの参照画面”だけ。
  ロードされた他案件の記憶は背景情報であってソースではない。
- **「良かれ」は行動でなく提案へ逃がす。** 入れた方が良いと思った要素は、勝手に
  置かず、チャットに「提案」として別枠で出してユーザーの判断を仰ぐ。
- 検証は各スキルの最終ゲートで行う（design-create/ds-editの **要素allowlist宣言 →
  最終差分監査**：宣言に無い要素が出ていたら削除かエスカレーション）。
- 出典タグは7種で統一する: **req（ユーザーの要件）/ ds（DSの既定）/ existing（このサービスの参照画面・node-id）/ inherent（そのオブジェクトに本質的）/ placeholder（仮置き・要差し替え）/ guess（推測・要確認）/ primitive-def（プリミティブ/基底定義）**。

### OOUI系（オブジェクトのビューとして起こす・出典ゲートと同型）

意図ベースの「OOUIで作れ」は効かない（生成時にAIは自分を正当化済みで違反が透明）。出典ゲートと同じく**宣言→機械的逆監査**で縛る。本ゲートはdesign-create側、ここは原則の宣言のみ。

- **新規画面/フローは「何のオブジェクト(名詞: メンバー/面談/目標等)の・どのビュー(collection一覧 / single詳細)か」を先に確定してから起こす。** 各フレーム=オブジェクトのビュー。
- **動詞はビュー上のモードレスなアクション**（chip / overlay / inline編集 / その場で完結し元ビューへ戻るmodal）にする。**独立画面・ステップ・ウィザードにしない。**
- **種別/立場/日程調整/定期/リマインドは“オブジェクト”でなくプロパティ・状態・アクション**として畳む。動詞・手続き・選択肢をオブジェクトに昇格させない。
- **退行禁止**: 「作成方法[今すぐ/単発/定期]を選ぶ」選択画面、Nステップウィザード、モード地獄（後戻り不可）。
- **逃がし弁あり**: 本質的に逐次なフロー（認証/決済commit/不可逆確認/初回オンボ等）は例外だが、**閉じた理由コード付きの宣言制**（黙って退行しない）。詳細ゲートと許可コードはdesign-create側。
- 詳細は design-create へ。design-createへ振り分けたら、まずビューインベントリ（オブジェクト×collection/single）を確定してから着手。

### 品質系（ビジュアル品質）
- **AIっぽい過剰な配色・装飾をしない。** 色は見栄えで選ばず、DSのセマンティック
  トークン＋実務の実カテゴリを根拠にする。
- **コントラストは WCAG AA 未満を出さない。** 「満たしている」と言う前に実測で確認する。
- **文字サイズは 14px を下限とする。** 独自に 14px 未満を作らず、DS の type style を使う。
- **複数案・別案は色やフォントの再着色でなく、構造・IA・見せ方から別物にする。**

### 進め方系（進め方・出力・文言）
- **新規ビルドは着手前にチャットでワイヤー（構造）を出して合意してから起こす。**
  手描きで進めない。
- **成果は Figma 上に直接書き戻す。** HTML モックやローカルファイルで出さない。
- **文言は自然な日本語で。** 見出し・キャッチに読点を入れすぎない、翻訳調・AI調は避ける。

## 判定フロー（ロードまでの手順）

入口の標準オーダーはこの1系統のみが正本。

0. **Figma作業はまずここ（`mikeneko-figma`）を通る。** いきなり他スキルやプラグインへ飛ばない。
1. **上の「判定（1問）」表で対象の単位を1問で割る**（既存全体 / 部品 / 画面 / 体験検証）。
2. **該当する自作スキルを1つだけロードする。** 複数該当は「対象の単位が一番大きいもの」を主にし、足りない部品だけ寄り道（グレーゾーンは「迷ったときの切り分け」を参照）。
3. **公式プラグインのスキルは振り分け対象にしない**（下請けとして各自作スキルが内部で呼ぶ）。
4. **共通の禁止事項（SOT）は全分岐で常に効く。** ロード先のスキルは、このSOTを再掲せず自分のゲートで検証する。
5. **design-createへ進む場合は、着手前にビューインベントリ（オブジェクト×collection/single）とページIA（オブジェクト単位のPage）を確定してから実装に入る。**

## 各スキルの最終ゲート対応表

4つの自作スキルが最後に通すゲートの俯瞰索引（承認／逆差分監査／OOUI逆監査／ページ編成逆監査／instance実体監査／完了ゲート）。詳細手順は各スキル本文に置く（ここは索引のみ・加算的）。

| スキル | ゲート | 発火タイミング | 合格基準（要旨） | 詳細§ポインタ |
|---|---|---|---|---|
| 全スキル共通 | 共通禁止事項（SOT） | 全工程 | 構造系/部品系/配置系/出典系/OOUI系/品質系/進め方系のいずれにも違反しない | 本ファイル「共通の禁止事項」 |
| ds-edit | 一括承認 | 設計→実装の手前 | 計画（変更/差替/配置）をユーザーが一括承認 | ds-edit 標準オーダー |
| ds-edit | 全件レビュー（逆差分監査） | 実装後 | 宣言allowlist外の要素が無い／別オブジェクト混入・動詞frame化が無い | ds-edit ⑤⑥ |
| ds-edit | instance実体監査 | 実装後 | 部品がinstance実体・slot利用をレイヤー種別/プロパティで構造確認（スクショ一致で合格にしない） | ds-edit ⑥ |
| component-design | 全組合せテスト | 設計後 | variant×boolean×swap×slotの全組合せが破綻なし（dead space無し） | component-design 検証 |
| component-design | 既存使用箇所への影響レビュー | 公開前 | 既存instanceへの破壊的伝播が無い | component-design 検証 |

| design-create | 最終差分監査（逆差分） | §8(e) | 宣言に無い要素が出ていない（default-deny） | design-create §8(e) |
| design-create | OOUI逆監査 | §8(f)・配線後に再実行 | 全frameが表Bへ写像済・線形ウィザード退行が無い・collection⇄single往復が成立（構造で裏取り） | design-create §8(f) |

| design-create | instance実体監査 | §8 | 部品がinstance実体・slot利用を構造確認 | design-create §8 |
