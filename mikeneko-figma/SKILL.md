# 自作Figmaスキルの入口（公式とは別系統）

Figma作業はまずここで **「どれを目指すか」を判定して、自作スキルを1つだけロード**する。
自分は `/mikeneko-figma` だけ覚えればよく、3つを毎回見比べて選ぶ必要はない。

**公式プラグインのスキルは振り分け対象に含めない。**
`figma-use` / `figma-generate-design` / `figma-generate-library` / `figma-generate-diagram` /
`figma-create-new-file` / `figma-code-connect` / `figma-swiftui` などは、各自作スキルが
内部で必要なときに呼ぶ「下請け（プラグインAPI層）」であって、ここで"目指す対象"にはしない。

## 共通の禁止事項（どのスキルに進んでも必ず守る）

詳細手順は各スキルに置く。ここは全作業に効く原則と、過去の事故から来る厳守事項。

### 構造・レイアウト
- **Spacer要素（余白埋め用の空frame/rectangle）は禁止。** 余白・間隔は全て
  auto-layout の gap（item spacing）と padding で制御する。固定の隙間ノードを
  挟んで見た目を合わせるのは不可。
- **場当たりの線・区切りで体裁を取り繕わない。** 「右側に線を引いただけ」の
  ような根拠のないデザインは禁止。各要素の意味に沿った構図で組む。

### 部品・コンポーネント
- **アイコンボタンを使うときはアイコンも必ず指定する。** アイコン未指定の
  プレースホルダのまま置かない。入れるアイコン（名前/ノード）を決めて実体をセットする。
- **部品はinstance実体で配置する（手描き偽装禁止）。** 生frame・外付けTEXTで
  部品に見せかけない。検証は各スキルの最終ゲートで行う（配置作業=ds-edit⑥／
  design-create§8のinstance実体監査でレイヤー種別/プロパティ/slot利用を構造確認、
  スクショ一致で合格にしない。部品を新規設計する場合はcomponent-designの全組合せ
  テスト＋既存使用箇所への影響レビューがこれに相当）。
- **マスターComp（メインコンポーネント）を編集して見た目を変えない。** 要素を消す・
  変種に合わせる時はインスタンス側で `visible=false`（行コンテナごと畳む）。

- **コンポーネントの強調・マーキングで fill/stroke を変えない。** 全インスタンスに
  伝播して事故になる。マーキングは外側のラッパーで囲んで行う。


### 配置・対象
- **置き場所のページは判定根拠を持って決める。** 既存の同種フレームがあるページ、
  または対象機能に対応するページに置く。該当が無い・複数候補で割れる場合は着手前に
  ユーザーへ確認し、自己判断で空いている所に置かない（「別ページに作って」は新規Pageを作る）。
- **node-id 指定はそのノードを in-place で編集する（複製・横並び禁止）。** Page を
  指定されたらその Page 上に作る。
- **勝手に新規コンポ・要素を作らない。** 作る前に既存DSに有るかを必ず確認する
  （検索やページ一覧の空振りは「無い」の証拠ではない。存在だけでなくslot/プロパティの
  capabilityまで読む）。重複作成しない。

### 要素の出典（スコープ膨張・文脈ブリード対策）

「良かれと思って足す」は本人にとって違反に見えないので、自制では止まらない。
**意図ベースの禁止ではなく、出典ベースのゲートで縛る。**

- **全要素は出典を持つ。出典の無い要素は置かない（default-deny）。** 各フィールド/
  要素/コピーは次のどれかに辿れること: **(a)ユーザーの要件 / (b)“このサービス”の
  参照画面（node-id） / (c)DSの既定**。どれにも紐付かない要素は禁止。「フォームは
  指定されたフィールドが全てで、それ以外は無い」を既定にする（「普通こうだから足す」
  をしない）。
- **他案件の情報は出典にならない（アンチブリード）。** 記憶・文脈・別プロジェクトに
  出てくる要素を混ぜない（例: 古着サービスの「店主に呼んでほしい名前」を無関係な
  会員登録に足す、等）。出典になるのは“今回の要件”と“このサービスの参照画面”だけ。
  ロードされた他案件の記憶は背景情報であってソースではない。
- **「良かれ」は行動でなく提案へ逃がす。** 入れた方が良いと思った要素は、勝手に
  置かず、チャットに「提案」として別枠で出してユーザーの判断を仰ぐ。
- 検証は各スキルの最終ゲートで行う（design-create/ds-editの **要素allowlist宣言 →
  最終差分監査**：宣言に無い要素が出ていたら削除かエスカレーション）。

### ビジュアル品質
- **AIっぽい過剰な配色・装飾をしない。** 色は見栄えで選ばず、DSのセマンティック
  トークン＋実務の実カテゴリを根拠にする。
- **コントラストは WCAG AA 未満を出さない。** 「満たしている」と言う前に実測で確認する。

- **文字サイズは 14px を下限とする。** 独自に 14px 未満を作らず、DS の type style を使う。
- **複数案・別案は色やフォントの再着色でなく、構造・IA・見せ方から別物にする。**


### 進め方・出力・文言
- **新規ビルドは着手前にチャットでワイヤー（構造）を出して合意してから起こす。**
  手描きで進めない。
- **成果は Figma 上に直接書き戻す。** HTML モックやローカルファイルで出さない。

- **文言は自然な日本語で。** 見出し・キャッチに読点を入れすぎない、翻訳調・AI調は避ける。


## 判定（基本は1問で割れる）

| やること | ロードするスキル |
|---|---|
| **既存**のものに手を入れる（restyle / 変数・テキストスタイル適用 / 生frame→component置換 / レビュー） | **`figma-ds-edit`** |
| **新規**に **部品1個**を起こす／**既存部品のproperty軸・sizingを大改修**する（ボタン/入力/バッジ/チップ等。variant・boolean・swap軸やslotの設計、全組合せテスト） | **`figma-component-design`** |
| **新規**に **画面・フロー丸ごと**（要件から既存DSの上に起こす＋プロトタイプ配線まで） | **`figma-design-create`** |

判定したら、そのスキルの鉄則・標準オーダーに従って進める。

## 迷ったときの切り分け（グレーゾーン）

- **既存frameの中身を直す** ＝ edit。 **新しいframe/画面を起こす** ＝ design-create。
- restyleしながら既存部品を別のDS部品に **差し替える** ＝ edit（差し替え先は既存DS部品をinstance化）。
- 既存部品の **property軸を大改修** する／ **新規に部品を起こす** ＝ component-design。
- design-createの途中で「**DSに無い部品が要る**」と判明 ＝ いったんcomponent-designへ寄り道して
  先に部品を設計してから画面作成に戻る（手描き偽装で代用しない）。
- 複数該当するなら **「対象の単位（既存全体 / 部品 / 画面）」が一番大きいもの** を主にして、
  足りない部品だけ寄り道する。
